kafka 101 · part 1

Three roles. One pipe in the middle.

Producers never talk to consumers directly. The broker stores everything, and consumers pull at their own pace.

4 / 22
kafka 101 · part 2

A topic is an append-only log. Nothing more.

retention window — not a queue, a log
This is why Kafka is cheap to replay: rewind the cursor, get every message again. The broker never "deletes on read".
5 / 22
kafka 101 · part 3

Partitions — one topic, sharded across brokers.

Partition is the unit of parallelism (more partitions = more consumers) and of order (order is per-partition, not topic-wide). Key → hash → partition.

6 / 22
kafka 101 · part 4

Consumer groups — scale a service horizontally.

rebalance

All pods share group.id. Kafka assigns partitions to pods so every message is handled exactly once per group. Adding or losing a pod triggers a rebalance.

7 / 22
kafka 101 · part 5

Fan-out — many teams, same topic, independent offsets.

every group — each at its own pace. this is why we need span links, not just parent/child.
One publish → N consumers, each with its own cursor. Parent/child tree can't model that; span links can.

The producer, by hand.

w := &kafka.Writer{
    Addr:         kafka.TCP("localhost:9092"),
    Topic:        "orders",
    Balancer:     &kafka.LeastBytes{},
    BatchSize:    100,
    BatchTimeout: 10 * time.Millisecond,
    RequiredAcks: kafka.RequireAll,
    Async:        false,
}
defer w.Close()

err := w.WriteMessages(ctx, kafka.Message{
    Key:   []byte(orderID),
    Value: payload,
})
if err != nil {
    log.Printf("publish: %v", err)
}

What's missing

  • no tracing — trace ID dies at the broker
  • no metrics — hand-roll Prometheus counters
  • no health — implement /health yourself
  • no graceful shutdown — wire up SIGTERM
  • no retries — beyond the writer's basics

~25 lines for one publish. Multiply by every service.

12 / 22
otel, briefly

Propagators carry what. Carriers carry how.

Propagator — the format

  • W3C TraceContext — traceparent · default
  • Baggage — user-defined k/v
  • B3 — Zipkin-compatible
  • Jaeger · Composite — stack several

Carrier — the transport

  • propagation.HeaderCarrier — over http.Header
  • propagation.MapCarrier — over map[string]string
  • your own — satisfy the 3-method interface
type TextMapCarrier interface {
    Get(key string) string
    Set(key, value string)
    Keys() []string
}
Kafka headers aren't http.Header and aren't map[string]string — they're []kafka.Header. So we implement the interface over that slice ourselves. That's the next slide.

without GoFr · part 3

Hand-rolling OpenTelemetry.

Step 1 — Implement TextMapCarrier over Kafka headers

type headerCarrier []kafka.Header

func (c headerCarrier) Get(k string) string {
    for _, h := range c {
        if h.Key == k { return string(h.Value) }
    }
    return ""
}
func (c *headerCarrier) Set(k, v string) {
    *c = append(*c, kafka.Header{
        Key: k, Value: []byte(v),
    })
}
func (c headerCarrier) Keys() []string { /* … */ }

Step 2 — Inject on publish, extract on subscribe

// publish
ctx, span := tr.Start(ctx, "publish",
    trace.WithSpanKind(trace.SpanKindProducer))
carrier := headerCarrier(msg.Headers)
otel.GetTextMapPropagator().Inject(ctx, &carrier)
msg.Headers = []kafka.Header(carrier)

// subscribe
carrier := headerCarrier(msg.Headers)
parent := otel.GetTextMapPropagator().
    Extract(ctx, &carrier)
sc := trace.SpanContextFromContext(parent)
var links []trace.Link
if sc.IsValid() {
    links = []trace.Link{{SpanContext: sc}}
}
ctx, span := tr.Start(parent, "subscribe",
    trace.WithSpanKind(trace.SpanKindConsumer),
    trace.WithLinks(links...),
)
Span links, not just parent/child — one Kafka message can fan out to N consumer groups; parent/child can't model that, links can. We do both.

with GoFr

The same flow. Business logic only.

api-gateway

app := gofr.New()

app.POST("/order", func(ctx *gofr.Context) (any, error) {
    var o Order
    if err := ctx.Bind(&o); err != nil { return nil, err }
    msg, _ := json.Marshal(o)
    return "ok", ctx.GetPublisher().Publish(ctx, "orders", msg)
})

app.Run()

order-service

app := gofr.New()

app.Subscribe("orders", func(ctx *gofr.Context) error {
    var o Order
    if err := ctx.Bind(&o); err != nil { return err }
    msg, _ := json.Marshal(validate(o))
    return ctx.GetPublisher().Publish(ctx, "alerts", msg)
})

app.Run()
Tracing, metrics, health, graceful shutdown, consumer groups, SASL, batching — all already on. Configured by env var.

no magic, just code

Here are the 50 lines GoFr hides for you.

pkg/gofr/datasource/pubsub/kafka/tracing.go 

type headerCarrier []kafka.Header
func (c  headerCarrier) Get(k string) string   { /* … */ }
func (c *headerCarrier) Set(k, v string)       { /* … */ }
func (c  headerCarrier) Keys() []string        { /* … */ }

func startPublishSpan(ctx context.Context, topic string) (context.Context, trace.Span, []kafka.Header) {
    ctx, span := tr.Start(ctx, "kafka-publish",
        trace.WithSpanKind(trace.SpanKindProducer))
    carrier := headerCarrier(nil)
    otel.GetTextMapPropagator().Inject(ctx, &carrier)
    return ctx, span, carrier
}

func startSubscribeSpan(ctx context.Context, topic string, h []kafka.Header) (context.Context, trace.Span) {
    parentCtx := ctx
    var links []trace.Link
    if len(h) > 0 {
        c := headerCarrier(h)
        ex := otel.GetTextMapPropagator().Extract(ctx, &c)
        if sc := trace.SpanContextFromContext(ex); sc.IsValid() {
            parentCtx = ex                                 // child → same trace ID
            links = []trace.Link{{SpanContext: sc}}        // link → fan-out aware
        }
    }
    return tr.Start(parentCtx, "kafka-subscribe",
        trace.WithSpanKind(trace.SpanKindConsumer),
        trace.WithLinks(links...))
}

You could write this. You don't have to. Same pattern in nats, google, sqs.
